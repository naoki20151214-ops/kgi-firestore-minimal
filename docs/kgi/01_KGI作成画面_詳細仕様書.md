1. この文書の目的  
  
この文書は、PDCAアプリにおける「KGI作成画面」の詳細仕様を定義するための文書である。  
  
特に、次の点をブレなく固定することを目的とする。  
  
- KGI作成画面の役割  
- 画面遷移の流れ  
- AIとLogicの役割分担  
- 入力項目と保存項目  
- 追加質問の出し方  
- KGI元データ確認の意味  
- KGI本体とKGI達成の判定条件の生成ルール  
- エラー時や失敗時の挙動  
  
この文書は、今後Codexへ修正依頼を出すときの基準とする。  
  
2. この画面の役割  
  
KGI作成画面の役割は、ユーザーのぼんやりした目標を、今期のKGIとして成立する達成状態へ具体化することである。  
  
この画面では、いきなりKPIやタスクを作らない。  
  
まず先に、  
「この期限の中で、どんな状態なら成功と言えるのか」  
を定義することに集中する。  
  
つまり、この画面の役割は  
「目標の原石を、解釈ズレの少ないKGIへ磨くこと」  
である。  
  
3. この画面で扱うKGIの定義  
  
この画面で扱うKGIは、  
「ユーザーの叶えたい未来に近づくために、この期間で達成すると決める成果状態」  
である。  
  
重要ルール:  
- KGIは時間で区切られた今期の到達点である  
- KGIはその期限の中で完結して判定できる必要がある  
- 期限をまたぐ条件は同じKGIに無理に入れない  
- 期限をまたぐ観測や反応確認は、必要なら次のKGI候補に分ける  
- KGIにはKPIやタスクを混ぜない  
  
4. この画面でやらないこと  
  
この画面では、次のことを主目的にしない。  
  
- 週何回やるかを決める  
- 毎日の行動計画を作る  
- KPIを確定する  
- タスク一覧を作る  
- 改善ログやボトルネック分析をする  
  
これらはKGI作成後の工程で扱う。  
  
5. 画面全体の基本フロー  
  
KGI作成画面は、以下の順で進む。  
  
ステップ1  
最初の2問入力  
  
ステップ2  
追加質問  
  
ステップ3  
KGI元データ確認  
  
ステップ4  
KGI確認・保存  
  
この流れは、原則として固定する。  
  
6. 各ステップの詳細  
  
6-1. ステップ1: 最初の2問入力  
  
目的:  
ユーザーの目標の原石を受け取る。  
  
入力項目:  
1. いつまでに達成したいですか？  
2. その期限の時点で、どんな状態になっていたらこの挑戦は成功と言えますか？  
  
仕様:  
- 回答は自由入力でよい  
- この時点では曖昧でもよい  
- ユーザーが簡単に書けることを優先する  
- ここでは正確なKGI文を求めない  
- 入力例を軽く見せてもよい  
- 入力後は「次へ進む」で追加質問へ進む  
  
許容される曖昧入力の例:  
- 副業で収益化したい  
- サイトを公開したい  
- 集客したい  
- 影響力をつけたい  
- FXで安定して勝てるようになりたい  
  
6-2. ステップ2: 追加質問  
  
目的:  
ステップ1で得た自由入力の曖昧さを減らし、KGIとして成立する達成状態へ近づける。  
  
最重要ルール:  
追加質問の目的は、進め方を聞くことではない。  
追加質問の目的は、達成状態の解釈ズレを減らすことである。  
  
追加質問の出し方:  
- 1問ずつ表示する  
- 基本は選択式にする  
- 必要な場合だけ「その他」で短い自由入力を許可する  
- 質問数は必要最小限にする  
- ただし曖昧さが残るなら複数問続けてよい  
- 最大質問数の上限は持つが、1問で打ち切らない  
  
質問生成の役割分担:  
Logicの役割:  
- 次に埋めるべき論点を決める  
- 質問数上限を管理する  
- 履歴を保存する  
- AI失敗時にフォールバックする  
  
AIの役割:  
- 現在までの文脈を読む  
- 次の論点に対して自然な質問文を作る  
- その質問に合う具体的な選択肢を作る  
- 文脈に応じて質問文と選択肢を変える  
  
追加質問で先に聞くべき内容:  
- 何をやるのか  
- どんなジャンルか  
- どんな形の成果物か  
- 何で収益が発生する想定か  
- 公開 / 完成 / 収益化の定義  
- 最低公開ライン  
- 必須条件か理想条件か  
- 品質条件  
- そのあと必要なら誰向けか  
  
追加質問で禁止すること:  
- 抽象的な自己分析質問  
- 前提不足のまま「誰向けか」「何を提供するか」を聞くこと  
- 同じ固定質問文を毎回使うこと  
- 毎回同じ固定選択肢を並べること  
- KPIやタスク寄りの質問を先に聞くこと  
  
悪い質問例:  
- あなたにとって価値ある状態とは何ですか？  
- 主に誰に何を提供できる状態なら達成ですか？  
- この目標が意味を持つのはどんな時ですか？  
  
良い質問例:  
- この副業はどの形にいちばん近いですか？  
- 今回作るものは、どの形に近いですか？  
- 収益は何で発生する想定ですか？  
- 公開とは、どの状態を指しますか？  
- 最低どこまでできたら公開してよいですか？  
  
6-3. ステップ3: KGI元データ確認  
  
目的:  
追加質問までの回答をもとに、AIが整理した「今期の具体案」をユーザーに確認してもらう。  
  
このステップの意味:  
いきなりKGI本体を作るのではなく、その前に  
「AIが今どう理解したか」  
を確認するための中間ステップである。  
  
ここが弱いと、KGI本体も弱くなる。  
  
このステップで表示する内容:  
- あなたが本当に叶えたい未来  
- 今回の期限で狙う範囲  
- 今回作るもの  
- 想定する読者  
- どんな価値を届けるか  
- 公開してよい最低ライン  
- 理想ライン  
- 今回は入れないもの  
- 次のKGI候補  
  
表示のルール:  
- 人間が読んで理解できる具体案にする  
- 構造化だけでなく narrative も持つ  
- 長すぎない短文で出す  
- スマホで読める長さにする  
- 箇条書きを優先する  
- 抽象語を減らす  
- 「追加確認中」などの逃げ文句は極力減らす  
- 仮説なら「仮説」と分かるように書く  
  
ここでのボタン:  
- だいたい合っている  
- 少し違うので質問に戻る  
  
挙動:  
- だいたい合っている: ステップ4へ進む  
- 少し違うので質問に戻る: ステップ2へ戻る  
  
6-4. ステップ4: KGI確認・保存  
  
目的:  
承認済みの元データから、KGI本体とKGI達成の判定条件を生成し、保存する。  
  
表示する内容:  
- KGI本体  
- KGI達成の判定条件  
- 整理した達成状態  
- 次のKGI候補（必要な場合）  
  
保存前にユーザーが編集できる項目:  
- KGI名  
- ゴール説明  
- 目標期限日  
- 説明レベル  
  
保存ルール:  
- source data 未承認なら保存不可  
- 保存時には source data も一緒に保持する  
- 自由入力 raw text も保持する  
- ただし自由入力をそのままKGI本文に流し込まない  
  
7. KGI元データの項目定義  
  
KGI元データには最低限次を持つ。  
  
- upperGoal  
  ユーザーが本当に叶えたい未来  
  
- kgiDeadline  
  今回の期限  
  
- rawSuccessStateInput  
  ユーザーの最初の自由入力  
  
- clarifiedSuccessState  
  AIが整理した達成状態  
  
- currentKgiScope  
  今回の期限の中で狙う範囲  
  
- concreteDeliverable  
  今回作る具体的な成果物  
  
- productShape  
  その成果物の形  
  例: ニュース一覧型、ブログ型、レビュー型、情報整理型、投資実践型  
  
- contentScope  
  載せる内容や扱う内容  
  
- audienceSummary  
  想定する読者または対象  
  
- valuePromise  
  届ける価値  
  
- minimumReleaseBundle  
  最低公開ライン  
  
- idealReleaseBundle  
  理想ライン  
  
- hardRequirements  
  必須条件  
  
- optionalRequirements  
  できれば入れたい条件  
  
- excludedFromCurrentKgi  
  今回は入れないもの  
  
- nextKgiSuggestion  
  次のKGI候補  
  
- sourceDataNarrative  
  人間向けの短文要約  
  
8. KGI本体の生成ルール  
  
KGI本体は、  
「この期間で達成する成果状態」  
として自然な日本語で書く。  
  
ルール:  
- 何をするかではなく、どんな状態を達成するかで書く  
- KPIやタスクを混ぜない  
- 進め方を書かない  
- 時間をまたぐ条件を入れない  
- 可能な限り具体的にする  
- 意味が通る自然な日本語にする  
  
悪い例:  
- 週3回投稿して収益化を目指す  
- ログを残して改善する  
- ボトルネックを特定する  
  
良い例:  
- 2026年7月3日までに、AI初心者向けの情報整理サイトを独自ドメインで公開し、第三者が閲覧できる状態を作る  
  
9. KGI達成の判定条件の生成ルール  
  
KGI達成の判定条件は、  
「何がそろっていたら達成とみなすか」  
を箇条書きで示す。  
  
ルール:  
- 解釈ズレが起きにくいこと  
- 達成状態の判断条件だけを書く  
- KPIや日々の運用ルールは入れない  
- できるだけ観測可能な条件にする  
  
例:  
- 独自ドメインで第三者が閲覧できる  
- トップページで記事一覧が見やすい  
- 5カテゴリ以上ある  
- 20記事以上ある  
- 初心者にも分かるやさしい表現になっている  
  
10. 期限をまたぐ条件の扱い  
  
重要:  
KGIはその期限の中で完結して判定できなければならない。  
  
したがって、次のようなものは同じKGIに無理に入れない。  
- 公開後7日以内に流入確認  
- 公開後14日以内に検索流入確認  
- 3ヶ月連続で月次プラス  
- 公開後の収益確認  
  
これらは必要なら次のKGI候補として切り出す。  
  
11. FXや株式投資などの自己運用型の扱い  
  
自己運用型は、公開型や情報発信型と根本的に違う。  
そのため、同じ質問テンプレを使ってはならない。  
  
自己運用型で禁止する質問:  
- 誰に届けたいか  
- 公開してよい最低ライン  
- 読者は誰か  
- 初期流入  
- 収益導線  
- サイトやコンテンツ前提の質問  
  
自己運用型で聞くべき内容:  
- 何をもって成功とするか  
- 連続月次プラスなどの集計定義  
- 対象市場や手法  
- 安定の定義  
- 除外条件  
- 今回の期限内で完結する成功判定  
  
12. AIとLogicの役割分担  
  
Logicの役割:  
- 画面遷移  
- ステップ管理  
- 質問数上限管理  
- 次に埋める論点の決定  
- 保存処理  
- エラー処理  
- 処理中表示  
- 多重押下防止  
- AI失敗時のフォールバック  
  
AIの役割:  
- 自由入力の意味解釈  
- 目標ジャンル判定  
- 追加質問の質問文生成  
- 追加質問の選択肢生成  
- KGI元データの具体化  
- sourceDataNarrative の生成  
- KGI本体の生成  
- KGI達成の判定条件の生成  
- 次のKGI候補の生成  
  
13. UIルール  
  
- スマホ最優先  
- ボタン押下中は必ず「処理中...」表示  
- 無反応に見える状態を作らない  
- エラー時は何が起きたか表示する  
- 追加質問は1問ずつ表示  
- 選択肢はラジオボタン  
- その他選択時のみ自由入力  
- 「KGI達成の判定条件」という文言を使う  
- 難しい言葉を避ける  
- 長文は折りたためるようにしてよい  
  
14. 保存データ  
  
保存すべき主な項目:  
- upperGoal  
- kgiDeadline  
- rawSuccessStateInput  
- clarifiedSuccessState  
- ambiguityPointsInitial  
- ambiguityPointsResolved  
- ambiguityPointsRemaining  
- followUpQuestionHistory  
- followUpAnswerHistory  
- aiKgiSourceData  
- sourceDataConfirmed  
- sourceDataEdited  
- kgiStatement  
- kgiSuccessCriteria  
- nextKgiSuggestion  
  
15. 固定ルール  
  
- Logicで抽象質問をしない  
- 前提不足のまま誰向けかを聞かない  
- 自由入力をそのままKGI本文に混ぜない  
- KGIの中にKPIやタスクを混ぜない  
- 期限をまたぐ条件を同じKGIに入れない  
- AI質問生成失敗時はフォールバックに切り替える  
- ユーザーに無反応に見える状態を作らない  
- KGI元データ確認を省略しない  
  
16. バージョン管理ルール  
  
- 大きな思想変更は minor を上げる  
- 軽微修正は patch を上げる  
- 文書タイトルは固定  
- Version と更新履歴を更新する  
- 情報源にアップした版を基準にする  
  
バージョン例:  
- 0.1.0 初版ドラフト  
- 0.2.0 KGI元データ確認を大幅強化  
- 0.2.1 UI文言調整
